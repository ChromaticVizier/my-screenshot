# 浏览器截图与录屏插件技术调研报告

## 1. 核心实现原理

浏览器截图和录屏插件的实现主要依赖三类底层技术方案，根据应用场景的不同，采用不同的技术组合。

### 1.1 截图方案的核心原理

| 技术方案 | 核心原理 |
|---------|---------|
| html2canvas | 遍历DOM结构，将HTML和CSS样式逐层解析并绘制到Canvas画布上，最终导出为图片 |
| WebRTC + getUserMedia | 通过`navigator.mediaDevices.getDisplayMedia()`获取屏幕视频流，捕获当前显示内容 |
| 浏览器原生API/CDP | 利用Chrome DevTools Protocol直接与浏览器渲染引擎通信，逐帧捕获画面 |
| Chrome 扩展 API（`chrome.tabs.captureVisibleTab`） | 浏览器扩展专属能力，由浏览器进程直接对当前标签页可视区域做一次原生渲染快照，返回 dataUrl |

理论上和直觉上来讲，html2canvas能获得最好的效果，但html2canvas模式的核心问题是性能与样式兼容性。当页面图片过多时响应会明显变慢，且对复杂CSS属性（如`filter`、`backdrop-filter`）和部分SVG效果支持不理想，可能会有错位现象。

WebRTC模式通过调用系统的屏幕共享选择器，用户需授权选择"整个屏幕"、"应用窗口"或"Chrome标签页"。此模式生成的图片质量最高，能捕获硬件加速渲染的内容，但必须在HTTPS或localhost环境下运行，主流的插件几乎都是这种方式实现的。

awesome的业务流程是：截图后先传到它的服务器上，然后再做处理。我测试时就截图后直接开一个下载任务。claude在实现、测试后推荐的：

`chrome.tabs.captureVisibleTab` 是浏览器扩展独有的方案，仅在 MV2/MV3 扩展上下文中可用。调用时由浏览器进程直接渲染当前活动标签页的可视区域并产出 PNG/JPEG，全程无任何用户交互弹窗（区别于 WebRTC 的屏幕共享选择器），且不受页面 CORS、HTTPS 限制。主要优势：

- 零交互体验：用户点击插件按钮即出图，符合「一键截图」的产品预期（待定）；
- 真实渲染质量：底层走浏览器渲染管线，画质等同 WebRTC，包含硬件加速合成结果；
- 轻量无依赖：不需要 `html2canvas`（数百 KB）或 `getDisplayMedia` 流转 Canvas 的额外开销；
- 权限最小化：配合 `activeTab` 权限，仅在用户主动触发时临时授权当前页，无需常驻 host 权限；
- 天然适配扩展架构：必须在 background service worker 中调用，刚好与 popup → background 的消息架构契合，便于后续叠加「延迟截图」「区域截图」等基于同一捕获能力的功能。

局限：仅能截取当前可视区域，无法用于长截图（需配合滚动拼接）或跨标签页录制；且对 `chrome://`、Chrome Web Store 等内部页禁止截图，需在业务层做拦截。

### 1.2 录屏方案的核心原理

| 技术方案 | 实现方式 | 优势 | 劣势 |
|---------|---------|------|------|
| MediaRecorder API | 捕获`getDisplayMedia`返回的`MediaStream`流，通过`MediaRecorder`编码为WebM/MP4 | 原生支持，实现简单 | 输出格式有限，压缩可控性差 |
| CDP + FFmpeg | 通过CDP逐帧截图，再调用FFmpeg编码成视频 | 质量高，支持多格式（MP4/MOV/AVI） | 依赖FFmpeg，资源消耗大，开发难度高 |
| Service Worker代理 | 拦截网络请求和渲染指令进行录制 | 可录制WebGL/Canvas内容 | 实现复杂，兼容性差 |

MediaRecorder是最常见的轻量级方案。将`getDisplayMedia`获取的`MediaStream`传入`MediaRecorder`，设置`mimeType`（如`video/webm`）和帧率后，通过`ondataavailable`事件接收视频数据块，最终生成Blob文件。

CDP方案特别复杂。它直接通过CDP协议监听浏览器的渲染帧，配合FFmpeg将帧序列编码为目标格式。由于不依赖窗口对象，它甚至可以在无头模式下稳定录制，且能自动跟随新开标签页。这种方式能实现超高帧率录制，但需要FFmpeg依赖，可能会比较重。另外，涉及ffmpeg视频编码的东西开发难度都会特别高，可能遇到一些奇怪的问题。

经测试，MediaRecorder API方案很难支持MP4格式录制，即使后期考虑转向ffmpeg也必须解决捕获问题。

### 1.3 长截图（滚动截图）的实现机制

长截图主要由三种方式实现：

1.  自动滚动拼接法（GoFullPage、FireShot）：插件控制页面滚动，每滚动一段距离截图一次，最后通过算法将多张图片无缝拼接。这种方法对页面布局稳定要求高，固定定位元素容易有残影。
2.  视口扩展法（Puppeteer方案）：临时将浏览器视口高度设置为`document.body.scrollHeight`，渲染完整页面后一次性截图。`brosh`工具支持`--height -1`参数实现此功能，较简单。
3.  增量渲染法：通过CDP指令逐屏渲染并捕获，避免一次性加载超大页面导致内存溢出（这种方法我目前没找到开源仓库，暂时没法评价操作难度）。

实际选型：在浏览器扩展（MV3）环境下，方案 2 不可直接使用（扩展无法修改外部 viewport），方案 3 需要 `chrome.debugger` 权限并会在页面顶部弹出"正在被调试"横条，体验差且需额外权限。因此本项目采用改良版滚动拼接法：

截图源：复用 `chrome.tabs.captureVisibleTab`，每滚动一屏截一次。
残影治理：滚动前由 content script 扫描所有 `position: fixed / sticky` 元素，将其临时改为 `visibility: hidden`（仅首屏保留首次出现的导航条），截完后恢复。这能解决调研中提到的"固定元素残影"问题。
限频处理：MV3 中 `captureVisibleTab` 有约 2 次/秒的频率限制，每次滚动后需 `await sleep(350ms)` 再截图。
拼接方式：在 background 的 OffscreenCanvas 上按 `devicePixelRatio` 等比拼接，处理最后一屏可能与前一屏重叠的部分（用 `scrollY` 实际位置而非"屏数 × 屏高"对齐）。
优势：不依赖额外权限、不引入新弹窗、单文件实现且可与「可视区域」共享底层捕获能力。

经测试可以稳定截图，效果和awesome差不多。

### 1.4 选区截图的实现机制

| 技术方案 | 思路 | 评价 |
|---------|------|-----|
| `getDisplayMedia` + Canvas 裁剪 | 用屏幕共享流抓帧后裁剪 | 体验差（弹共享选择器）、HTTPS 限制 |
| `html2canvas` 抓 DOM 后裁剪 | 解析 DOM 重绘到 Canvas，再按选区裁剪 | 性能差、CSS 兼容差，已在 1.1 节排除 |
| `captureVisibleTab` + content script 选区 + Canvas 裁剪（本项目选用） | 注入遮罩让用户拖拽选区 → 整屏截图 → 按坐标裁剪 | 体验最佳、零依赖、与可视区域复用底层 |

实现要点：

- 选区交互：content script 在页面上注入一层 `position: fixed; inset: 0; z-index: 2147483647` 的遮罩，监听 `mousedown / mousemove / mouseup`，用半透明矩形可视化选区，按 `Esc` 取消。
- 坐标传递：拿到选区的 `{ x, y, width, height }`（CSS 像素，相对于视口左上角）后，通知 background。
- 裁剪：background 调 `captureVisibleTab` 拿整屏 dataUrl，转 Blob → `createImageBitmap` → 绘制到 OffscreenCanvas（按 `devicePixelRatio` 放大坐标），导出指定区域为 PNG Blob。
- 下载：用 `URL.createObjectURL(blob)` + `chrome.downloads.download` 触发下载（service worker 中 `URL.createObjectURL` 自 Chrome 110+ 已支持，本项目目标 Chrome 已满足）。

经测试也能实现基本功能，具体样式待优化。

## 2. 优劣势分析

### 2.1 截图方案对比

| 特性 | html2canvas (DOM解析) | WebRTC (屏幕流) | CDP (开发者协议) | captureVisibleTab (扩展API) |
|------|:---:|:---:|:---:|:---:|
| 图片质量 | 中等（依赖CSS解析） | 最高（真实渲染） | 最高（无损捕获） | 最高（真实渲染） |
| 性能开销 | 高（大页面卡顿） | 低 | 低 | 极低（一次原生快照） |
| HTTPS要求 | 无 | 必须 | 无 | 无 |
| 跨域限制 | 有（CORS） | 无 | 无 | 无 |
| 用户交互 | 无 | 需选择共享源 | 无 | 无 |
| 适用场景 | 局部元素、移动端 | 全屏、应用窗口 | 自动化测试、脚本 | 扩展内可视区域截图 |

### 2.2 录屏方案对比

| 特性 | MediaRecorder API | CDP + FFmpeg | WebRTC 录制 |
|------|:---:|:---:|:---:|
| 输出格式 | 有限（WebM为主） | 丰富（MP4/MOV/AVI） | 取决于实现 |
| 音轨支持 | 支持（需采集麦克风/系统音频） | 不支持（纯画面） | 支持 |
| 资源占用 | 中等 | 较高（FFmpeg编码） | 中等 |
| 无头模式 | 不支持（需UI） | 支持 | 不支持 |

目前选型：popup 在用户手势栈里 `chrome.tabCapture.getMediaStreamId` 拿 streamId（不传 consumerTabId），background 拉起一个屏幕外的扩展中转窗口里跑 `getUserMedia` + MediaRecorder，输出 webm 走 `chrome.downloads`；网页进程自己跑 MediaRecorder 输出的 webm 在 Chrome 内置播放器无法播放，搬到扩展自有 origin 的中转窗口才正常。区域录制额外把源 video track 经 `MediaStreamTrackProcessor` → `VideoFrame.visibleRect` 帧级裁剪 → `MediaStreamTrackGenerator` 输出新 track，再喂 MediaRecorder；走 canvas.captureStream 中转的版本因 frame timestamp 缺失同样无法播放。

### 2.3 主要难总结

- 音频录制复杂度高：与采集画面不同，捕获系统内部音频（非麦克风）涉及复杂的路由处理（如`chrome.tabCapture` API）。目前多数纯网页录屏插件不支持内部音频，或存在音画不同步问题。
- 移动端兼容性差：移动端浏览器不支持`getDisplayMedia`，主要依赖`html2canvas`。由于移动端WebView性能限制，截图速度较慢，且必须禁止页面缩放（`user-scalable=no`）以避免UI错位。
- Electron等桌面框架适配：在Electron环境中，`getDisplayMedia`行为与标准浏览器不同。开发者需要自行通过`desktopCapturer`获取屏幕源ID，并传入插件。
- 另外，高质量的开源项目比较难找，github上很多十几年前的项目，其中用的一些api在新版chrome中都已经不支持了，所以获得外部参考的机会少。
