# 整页截图主流程分析

## 一、总体架构

整页截图有**四套实现**，优先级由高到低：

| 优先级 | 模式 | 开关 | 入口 |
|--------|------|------|------|
| 1 | 逆向还原模式 | `reverseRestoreMode` | `captureFullPageReverse.ts` |
| 2 | CDP 模式 | `useCdpForFullPage` | `captureCdp.ts` |
| 3 | 激进隐藏模式 | `aggressiveHideMode` | `captureFullPageAggressive.ts` |
| 4 | 标准模式（默认） | — | `capture.ts handleCaptureFullPage` |

背景调度逻辑在 `src/background/index.ts` 中按上述优先级分发；本文聚焦后三套滚动拼接实现（CDP 模式一次性截全页、原理不同，略去）。

---

## 二、标准模式主流程

> 文件：`src/background/handlers/capture.ts` `handleCaptureFullPage`

```
┌──────────────────────────────────────────────────────────┐
│ 0. getCapturableActiveTab()  确认可截图的 tab            │
│ 1. resolveFrameTarget()      确定注入目标 frame          │
│ 2. preparePage()             锁定 overflow，探测主 scroller│
│    └─ 注入 freeze style（禁 smooth/动画）                │
│ 3. flattenOversizedModals()  摊平超屏 fixed iframe 弹窗  │
│    └─ freezeFlattenedModals() 阻止页面 JS 关闭弹窗        │
│ 4. scrollToY(0)  滚回顶部                                │
│ 5. captureVisibleTab()       首帧（保留顶栏/弹窗原样）   │
│                                                          │
│ 6. 若页面超过一屏：                                       │
│    a. hideFixedElements()    一次性 display:none fixed/sticky│
│       hideFixedElementsExcludeFrame()  子 frame 模式额外清主 frame│
│    b. detectAndHidePseudoSticky()  探测 JS 伪 sticky      │
│    c. measureTopHeaderBottom() + measureContentTopReservedSpace()│
│       → contentOffsetY：干净帧整体下移，给顶栏带让位    │
│    d. measureScrollMetrics()  重测隐藏后的文档高度        │
│    e. LOOP:                                              │
│       scrollToY(targetY)                                 │
│       waitForDynamicContent() 等懒加载稳定               │
│       measureScrollMetrics()  重读 scrollTop/Height      │
│       rehideFixedElements()   补隐藏 SPA 重新挂载的顶栏  │
│       measureScrollMetrics()  capture 前最终重读         │
│       captureVisibleTab()     截一帧                     │
│       → 终止：stall×4 或 scrollY+step ≥ totalHeight      │
│                                                          │
│ 7. restoreFixedElements() + restorePage()  恢复页面状态  │
│ 8. stitchToBlob()  OffscreenCanvas 拼接                  │
│ 9. downloadImageBlob()  下载                             │
└──────────────────────────────────────────────────────────┘
```

### 关键参数
- `stepHeight = viewportHeight × (1 - overlapRatio)`，默认重叠 5%，可在设置页调整（0~50%）
- `captureVisibleTab` 限频 600ms/次（Chrome 限制约 2 次/秒）
- 动态内容等待：80ms 轮询，连续 3 次 scrollHeight + 图片完成数不变即算稳定，上限 1500ms

## 三、特殊页面处理机制

### 3.1 SPA 内部滚动容器（知乎、网易邮箱等）

**问题：** SPA 常用 `<div style="height:100vh; overflow:auto">` 做主体滚动，`window` 完全不可滚，直接 `window.scrollTo` 只能拍首帧。

**处理方式：** `preparePage()` 中多维度评分自动识别主 scroller：
- 可滚动溢出量（`scrollHeight - clientHeight`）
- 视口覆盖面积（宽≥15% vw、高≥25% vh）
- 文本量
- 语义类名正则（main/content/chat/list 等）
- 居中程度、深度惩罚

找到后打 `data-my-screenshot-scroller="1"` 标记，后续所有注入函数通过该标记同步找到同一元素。

**评估：**
- ✅ 覆盖了常见单内容区 SPA（知乎、Confluence、网易邮箱）
- ⚠️ 评分模型在"多个并列可滚区"（三栏布局）场景容易误判；语义正则和面积权重可能把侧边栏排行榜识别为主体
- ⚠️ `overflow:hidden` 容器靠 JS 赋 `scrollTop` 也能滚动（网易邮箱即此类），代码已处理，但依赖 `overflowY !== "visible"` 这一判断，对特殊布局可能漏判

---

### 3.2 顶栏吸顶重复（几乎所有门户/SPA）

**问题：** `position:fixed/sticky` 顶栏在每一帧截图中都在同一视口坐标，拼接后每屏出现一次 → 长图里重复 N 次。

**处理方式：**
1. 首帧保留顶栏原样（让长图顶部有完整导航栏）
2. 截完首帧后一次性 `display:none` 所有 `fixed/sticky` 元素
3. 计算 `contentOffsetY = headerBottom - reservedTop`，干净帧整体下移，使原本被顶栏遮住的标题等内容完整露出
4. 每帧前调 `rehideFixedElements()` 补隐藏 SPA scroll 回调里重新挂载的顶栏

**评估：**
- ✅ 设计合理，首帧顶栏 + 干净正文的组合方案消除了绝大多数重复
- ⚠️ `measureTopHeaderBottom` 判定依据是"贴视口顶（≤2px）、宽度≥视口50%、高度≤视口60%"；极窄顶栏（如仅含 logo 的 4px 栏）或宽度不足50%的侧边吸顶栏会漏判，`contentOffsetY=0` 导致首帧正文被遮
- ⚠️ `measureContentTopReservedSpace` 扫描前 4000 个节点取"最上沿可见文本/图片"；页面顶部若有 1px 透明装饰图，reservedTop 会被拉低到 0，下移量虚增，长图顶部出现一段多余空白

---

### 3.3 JS 伪 sticky（Confluence 顶栏等）

**问题：** 顶栏用 JS 监听 scroll + 修改 `transform/margin/top` 来模拟跟随，`computed position` 仍是 `static/relative`，`hideFixedElements` 扫不到。

**处理方式：** `detectAndHidePseudoSticky()` 物理探测：
1. 记录所有首屏可见元素的绝对位置 `(rect.top + scrollY)`
2. 向下滚动 200px
3. 重测：若元素绝对位置漂移量 ≥ 实际滚动量×0.7，判定为"跟随视口" → `display:none`
4. 滚回原位

**评估：**
- ✅ 有效覆盖 Confluence 等主流 SPA 的 JS 顶栏
- ⚠️ 探测本身会产生一次小范围滚动，对"滚动触发内容加载"的页面（如知乎 infinite scroll）可能误触懒加载，使页面内容提前加载到后续本不该出现的区域
- ⚠️ `FOLLOW_RATIO=0.7` 是固定阈值；吸附到整数像素的 CSS `top` 动效（漂移量非连续）可能漏判
- ⚠️ 探测前提是"首屏可见"。若伪 sticky 顶栏在 `scrollY=0` 时已经滚出视口（用户不在顶部触发截图），探测的候选集为空，直接返回 0，漏隐藏

---

### 3.4 含 iframe 的弹窗（有道字典"全部产品"等）

**问题：** `position:fixed` 的弹窗内嵌 iframe，整体超出首屏高度；隐藏后滚动时弹窗内容被裁断。

**处理方式：** `flattenOversizedModals()`：
1. 找到"含 iframe 且 bottom > viewportHeight" 的 fixed 弹窗
2. 把它改为 `position:absolute`，坐标换算到文档坐标系，让其底部能延伸到文档下方
3. 等待 800ms（允许 iframe reload）
4. `freezeFlattenedModals()`：装 `MutationObserver` + 吞噬鼠标/焦点事件，阻止页面 JS 把弹窗收回

**顺序约束：** flatten + freeze 必须在 `scrollToY(0)` **之前**执行。原因：很多 dropdown 监听 scroll 事件，一旦滚动就 `display:none` 自身，flatten 后位置已用文档坐标写入，MutationObserver 可以阻止 JS 的 display:none。

**评估：**
- ✅ 解决了核心问题，MutationObserver 冻结方案有效
- ⚠️ 仅处理"含 iframe"的超屏弹窗；不含 iframe 的纯文本超屏弹窗不在范围内，截图时可能被裁
- ⚠️ 800ms 等待是经验值；网速慢时 iframe reload 未完成就截图，首帧会出现空白 iframe
- ⚠️ `unfreezeFlattenedModals` 在 finally 里调用，若 tab 关闭或中途崩溃，MutationObserver 残留在页面 JS 上下文，不影响用户（页面关闭），但 `restoreFlattenedModals` 失败时弹窗 position 状态无法回滚

---

### 3.5 动态/懒加载页面（知乎 infinite scroll、虚拟列表）

**问题：** 滚到新位置后内容不立即渲染；某些 infinite scroll 只监听 `wheel` 事件而非 `scroll`，直接设 `scrollTop` 不触发加载。

**处理方式：**
1. `waitForDynamicContent()`：80ms 轮询 scrollHeight + 视口内图片完成数，稳定 3 次或超时 1500ms 才截图
2. `measureScrollMetrics()`：每帧截前再读一次 scrollHeight，动态更新总高度上限
3. `kickScrollListeners()`：stall 时派发 `scroll` + `WheelEvent` 事件戳醒懒加载

**评估：**
- ✅ 三层联动，基本覆盖主流懒加载模式
- ⚠️ `DYNAMIC_WAIT_MS=1500ms` 对网速极慢或渲染耗时长的页面仍可能截到空内容
- ⚠️ 虚拟列表（如 AG Grid、react-window）在滚动时会回收 DOM，图片完成数签名会突变，导致 stableCount 不断重置，几乎总是等到超时才截图，拖慢速度
- ⚠️ `kickScrollListeners` 派发的是合成事件，部分检测 `isTrusted` 的实现不响应

---

### 3.6 跨 frame 页面（Popo 文档、Confluence 嵌套 iframe）

**问题：** 用户手动用 picker 选了某 iframe 内的滚动区，截图流程需要精确注入该子 frame 并计算其在主 frame viewport 中的偏移，才能从全屏截图中裁出正确区域。

**处理方式：**
- `resolveFrameTarget()`：`chrome.webNavigation.getAllFrames` 枚举所有 frame，完全匹配 → 模糊匹配（同 origin + path 第一段）→ 回退主 frame
- `locateFrameOffsetInPage()`：同源递归文档树查找 iframe 的 `getBoundingClientRect`；跨域中间层取外层偏移作近似值；全部失败后兜底取"最大可见 iframe"
- `captureHeight` 夹紧到 `tabViewportHeight - frameOffsetY - captureY`，防止 slice 源高超出 bitmap 边界

**评估：**
- ✅ 偏移计算和高度夹紧逻辑周全
- ⚠️ 跨域 iframe 无法读取 `contentDocument`，偏移计算退化为外层 iframe 位置；若目标 iframe 在跨域中间层内还有偏移（如 padding/margin），会产生系统性偏差
- ⚠️ "最大可见 iframe" 兜底假设页面只有一个主 iframe；多个等大 iframe 的页面（如双栏对照文档）会随机命中

---

### 3.7 scroll-behavior:smooth 容器

**问题：** `scrollTop = y` 在带 `scroll-behavior:smooth` 的容器上返回的是动画初期值，远小于目标，导致截图位置错误。

**处理方式：**
- `scrollToY()` 临时覆盖 `scrollBehavior="auto"` 和 `scrollSnapType="none"`，赋值后立即还原
- 每帧 `scrollToY` 后 sleep 120ms，再用 `measureScrollMetrics` 读稳定 scrollTop
- `preparePage` 注入全局 freeze style：`html { scroll-behavior: auto !important }`

**评估：**
- ✅ 三层保障基本消除 smooth 问题
- ⚠️ `!important` freeze style 会覆盖用户自定义样式；极少数站点依赖 `scroll-behavior:smooth` 触发自身动画钩子，freeze 后动画不执行，可能影响截图内容

---

### 3.8 激进隐藏模式

**机制：** 在标准模式基础上，截图前先"隔离"主 scroller：把 scroller 祖先链以外的所有兄弟节点一律 `display:none`（`isolateScroller()`），再叠加 `hideFixedElements + detectAndHidePseudoSticky`。隔离后页面只剩 scroller 子树，无需首帧保留 + contentOffsetY 补偿，所有帧统一处理。

**评估：**
- ✅ 彻底消除顶栏重复，简化拼接逻辑
- ⚠️ 「内容分散在多个并列容器」的页面（词典主页、瀑布流等）会漏截 scroller 以外的内容
- ⚠️ 隔离会触发回流，scroller 位置可能变化，代码已用 `measureScrollerRect` 重测，但极端情况（回流引起字体加载、图片延迟）仍可能产生尺寸抖动

---

### 3.9 逆向还原模式

**机制：** 还原自某闭源插件的 DOM 操作：把 `fixed/sticky` 改为 `position:relative` 落回文档流（"就地静态化"），顶栏因此只在自身所在那一屏出现一次，无需 contentOffsetY 补偿。相邻帧固定重叠 200px（对齐闭源实测值）。每帧截前调 `reverseRestaticize()` 补处理 SPA 重新吸附的顶栏。

**评估：**
- ✅ 顶栏处理更自然（就地出现而非补偿移位），对复杂吸顶站点效果优于标准模式
- ⚠️ 固定重叠 200px 在视口高度较小（如 600px）时重叠比例达 33%，产生大量冗余帧，速度明显慢于比例重叠方案
- ⚠️ `position:relative` 回流后原 sticky 的空位消失，相邻内容可能上移，对布局依赖 sticky 占位高度的页面（如某些表格的吸顶表头）会引起内容错乱
- ⚠️ `reverseRestorePage` 依赖 inline style 快照还原；若截图期间页面 JS 也修改了相同元素的 inline style（SPA 常见），还原后状态可能不正确

## 四、图像拼接

> 文件：`src/background/utils/imaging.ts` `stitchToBlob()`

```
canvasH = totalHeight × devicePixelRatio
for each slice:
  dy = slice.scrollY × dpr          // 在长图中的 Y 偏移
  从 bitmap 裁出 [sourceX,sourceY,sourceWidth,sourceHeight]
  绘到 canvas [0, dy]
```

- 后绘覆盖先绘：重叠区域取"较新"帧，与滚到底的视觉一致
- 先填白底：避免 dpr 非整数时帧间 1px 透明缝隙在 JPEG / 某些查看器中显示为黑线

**评估：**
- ✅ 逻辑简洁，OffscreenCanvas 在 Service Worker 中无需 DOM
- ⚠️ 所有帧 bitmap 在内存中同时存在，极长页面（>20 屏）可能 OOM；无分批绘制机制
- ⚠️ `totalHeight × dpr` 超过浏览器 canvas 最大尺寸（Chrome 约 16384px）时 `new OffscreenCanvas` 静默失败，后续 `convertToBlob` 返回空数据，用户只会收到下载失败，无明确错误提示

## 五、综合问题总结

| 分类 | 问题 | 严重程度 |
|------|------|----------|
| 内存 | 超长页面所有帧 bitmap 同时在内存，无 OOM 保护 | 高 |
| Canvas 上限 | 超 16384px 的长图 OffscreenCanvas 静默失败 | 高 |
| 首帧遮挡 | `measureTopHeaderBottom` 对窄/非横向顶栏漏判，正文标题被遮 | 中 |
| 伪 sticky 漏判 | 非首屏触发截图时探测候选集为空 | 中 |
| 虚拟列表 | DOM 回收导致图片签名频繁变化，几乎总等到超时 | 中 |
| 固定重叠帧数多 | 逆向模式 200px 固定重叠在小视口下冗余帧多，速度慢 | 低 |
| iframe 跨域偏移 | 跨域中间层偏移计算只取外层，有系统性偏差 | 低 |
| SPA inline style 冲突 | 逆向模式还原时与 SPA JS 同写 inline style 可能状态不一致 | 低 |
| 探测滚动触发懒加载 | `detectAndHidePseudoSticky` 的 200px 探测滚动可能触发 infinite scroll | 低 |
