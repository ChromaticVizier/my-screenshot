# 录屏功能开发实录

记录从零实现「当前标签页录屏」与「区域录制（当前标签页）」的完整过程，重点是每一次方案选型的取舍与踩坑。

## 1. 总体目标

在录屏 tab 提供两个按钮。「当前标签页」是一键录制活动 tab 的整个视口，无须任何选择器；「区域录制」点击后由用户在页面上拖出一个矩形，松手开始录制，矩形边框留在页面上作为录制范围标识，控制栏显示在左下角。两者都通过控制栏停止后下载 webm 视频。

录制过程中如果用户跳转到其它 url（含 SPA 与真跳转），控制栏与边框都要持续可见且唯一。

## 2. 第一版：注入脚本里直接跑 MediaRecorder

最自然的 Chrome 扩展录屏架构：popup 在用户手势同步栈里调 `chrome.tabCapture.getMediaStreamId({ targetTabId, consumerTabId })` 拿到 streamId，发消息给 background，background 通过 `chrome.scripting.executeScript({ func })` 把一段自包含的「录制+控制栏 UI」函数注入到目标 tab 的网页进程，注入函数里用私有约束 `chromeMediaSource: "tab"` + `chromeMediaSourceId: streamId` 调 `getUserMedia` 拿到 MediaStream，扔给 MediaRecorder，停止时把 chunks 拼成 blob，再 base64 编码成 dataUrl 发回 background 走 `chrome.downloads.download` 下载。

写完跑通，文件能下载，但拖到 Chrome 内置播放器里进度条灰色、时间显示 0、播放按钮不可点。MediaRecorder 输出的 webm 缺 `Segment.Info.Duration` metadata 字段（crbug 642012），Chrome 内置 `<video>` 拒绝渲染控件。

## 3. 尝试 EBML 重写补 Duration

写了一个简化的 EBML 解析器，找到 Segment 容器，再找 Info 容器，在 Info 末尾追加一个 Duration 元素并改写 Info 的 size VINT。结果文件比之前更糟，整个画面都打不开。原因是 MediaRecorder 输出的 Info size VINT 通常用 1–2 字节定长写入，我用 8 字节定长重写后 Info 后续的所有元素字节偏移整体后移，Segment 里的 SeekHead/Cues 引用全部失效，播放器找不到 Tracks 直接报错。

接着改成「完整重新序列化 Segment」，把 Segment 的所有直接子元素读出来，对 Info 子元素列表做 upsert Duration，再用最小宽度 VINT 重新拼接。同时遇到 SeekHead 时直接放弃修补返回原 blob 以免破坏。本地测试仍然不能播：实测 Chrome 输出的 Info 长度变化后，Cluster 内部时间戳已经是空的，光补 Duration 字段不够。

到这一步意识到方向错了：问题不是 Duration 元素丢失这一个字段，而是 content script 所在的网页进程产出的 webm 整体元数据偏少，扩展自己代码层面修不动；得换 MediaRecorder 的运行环境。

## 4. 切换到中转窗口架构

参考之前屏幕截图功能的做法。Plasmo 0.90.5 不支持 offscreen document 入口，但已有的「中转扩展窗口」模式足够：让 background 用 `chrome.windows.create({ type:"popup", url:"popup.html?action=offscreenRecorder" })` 拉起一个尺寸 1×1、坐标在屏幕外（通过 `chrome.system.display.getInfo` 算出所有显示器的下边界再 +50px）的窗口，popup.tsx 入口根据 query 参数分流到 `OffscreenRecorder` 组件。

录制流程改为：popup 仍然在用户手势栈里申请 streamId，但**不传 consumerTabId**（让 streamId 可被任意扩展上下文消费）；background 收到 streamId、tabId 后做三件事：把控制栏 UI 的注入函数（精简过，只剩 DOM + 计时 + 按钮 + 消息转发）注入到目标 tab；把 streamId、文件名、麦克风/系统音频选项写入 `chrome.storage.local`；创建中转窗口。中转窗口加载完成后立刻读 storage、调 `getUserMedia` 消费 streamId、启动 MediaRecorder。停止时控制栏发 `record/stop` 给 background，background 广播 `recorder/stop` 给中转窗口，MediaRecorder.stop 触发 onstop，里头用 `URL.createObjectURL(blob)` + `chrome.downloads.download`（中转窗口是扩展自有 origin 页面，可调 chrome.downloads，blob URL 也在同一进程内有效），下载完发 `recorder/finish`，background 关闭中转窗口、清理会话。

视频在 Chrome 内置播放器里能正常播了。同样的 webm 格式在网页进程输出不行、在扩展自有页面输出可以——两个进程产出的 webm 元数据细节确实不同，但具体差在哪我没继续往下挖。

## 5. 跳转后控制栏残留的修复

切换 url 后旧 onMessage listener 跟着旧文档销毁，停止消息发不到任何地方，控制栏 DOM（如果还在新文档里）就会残留。两步处理：在 background 顶层注册一次 `chrome.webNavigation.onCommitted` 监听器（持久化挂在 service worker 顶层，被唤醒时 listener 自动恢复），录制中目标 tab 的主框架跳转完成后，从内存或 storage 恢复活跃 tabId 与 startTime，重新 executeScript 注入控制栏；停止时除了广播消息，也主动 executeScript 跑一个 `removeRecorderControlBar` 函数，按 `data-my-screenshot-recorder-bar` 属性把所有残留 DOM 都清掉，不依赖控制栏自身 listener。控制栏注入函数本身也做了防重复处理：进入时先清理同标记的 DOM，再挂新的，保证页面同一时刻只有一份。

## 6. 区域录制：选区交互

复用截图侧已有的 `pickSelection` 注入函数，加一个 `keepFrameAfterPick` 参数。截图模式松手后所有覆盖物移除（原有行为不变）；录制模式松手后保留一个红色边框 div，附带 `data-my-screenshot-region-frame` 属性。

由于注入函数是通过 `executeScript({ func })` 序列化的，函数体不能引用模块顶层变量，所以把 helper 内联进函数体或者通过 args 传参。我把它合并成单个 `pickSelection(args)`，截图侧调用补 `args: [{ keepFrameAfterPick: false }]`，录制侧传 `true`。同时加 `injectRegionFrame(rect)` 与 `removeRegionFrame()` 两个独立的注入函数，分别用于页面跳转后恢复边框、停止录制时清理边框。

popup 端新增 `startRegionTabRecording`：与整页录制共用 streamId 申请逻辑（同样不传 consumerTabId），只是发给 background 的消息类型变成 `record/startRegionTab`。background 在 handler 里先注入 picker 等用户拖完拿 rect，再走和整页录制完全相同的 `bootstrapRecorder` 启动主流程，区别仅是把 rect 写入 RECORDER_BOOT.region。streamId 有 10 秒有效期，从用户点开始到拖完通常 3–5 秒，够用。

webNavigation 跳转后既要重注控制栏也要重注红边框，这两件事在同一个 onCommitted 回调里串行执行。

## 7. 区域录制的视频源裁剪：先 canvas、再 WebCodecs

第一版用最直观的方式：中转窗口里建隐藏 `<video>` 接 tabStream，再建 canvas 按 region × dpr 计算裁剪框，`requestVideoFrameCallback` 每帧把 video 画到 canvas，最后 `canvas.captureStream()` 拿 video track 喂给 MediaRecorder。

下载下来的视频又是「Chrome 内置播放器进度条灰色、时间 0」。区别于整页录制时这条路径是工作的，唯一差异是视频 track 的来源——`canvas.captureStream()` 输出的 track 没有真实 frame timestamp 源（只有从 canvas 绘制时间推断），MediaRecorder 写 webm 的 Cluster timecode 就异常。试过把 `canvas.captureStream(30)` 的 fps 参数去掉、改用 `requestVideoFrameCallback` 与源帧严格对齐，都没改善。问题在 captureStream 这一层无法解决。

第二版换用 WebCodecs Insertable Streams（Chrome 94+）。把源 tab 的 video track 接到 `MediaStreamTrackProcessor`，拿到 `ReadableStream<VideoFrame>`，过一个 TransformStream，对每个 VideoFrame 用 `new VideoFrame(src, { visibleRect: { x, y, width, height }, timestamp: src.timestamp })` 做帧级裁剪，关键是显式保留原帧的 timestamp，再 pipe 到 `MediaStreamTrackGenerator({ kind:"video" })`，generator 自身就是一条 video track。VideoFrame 构造器对 visibleRect 是零拷贝重解释，性能开销很小。

用这条路径输出的 track 与原 tab track 在时间戳层面完全一致，MediaRecorder 写出的 webm 与整页录制路径同质，Chrome 内置播放器正常播放。

## 8. 还没做的事

mp4 输出（Chrome MediaRecorder 还不稳）、桌面/摄像头模式、控制栏暂停按钮在中转窗口里的状态同步（目前控制栏的暂停只发消息，UI 状态本地维护，没有 round-trip 校验）、中转窗口 1×1 尺寸在某些 Windows 主题下窗口装饰会让它仍占据一两个像素，目前靠挪到屏幕外回避，没有彻底隐藏。
